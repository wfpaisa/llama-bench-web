## Bug: el optimizador infla el KV cache ~4× en modelos híbridos SSM/Attention

**Raíz confirmada:** la fórmula del KV cache asume que TODAS las capas (`block_count`) generan KV cache. En modelos híbridos (Qwen3.5/3.6, Jamba, Zamba, MiniMax, Nemotron-H, Falcon-H1, Granite-Hybrid) solo una fracción de las capas son de atención; el resto son SSM/Mamba recurrentes con estado fijo que **no** escala con el contexto. Para tu Qwen3.5-9B (32 capas, `full_attention_interval=4`): solo 8 capas tienen KV → el cálculo está 4× inflado (8,54 GiB estimados vs ~2,1 GiB reales).

**Solución canónica (de llama.cpp, genérica):** leer una cadena de precedencia de claves GGUF para deducir cuántas capas son de atención:
1. `attention.recurrent_layers` (array de bool por capa) → `attnLayers = L − count(true)`
2. si no, `full_attention_interval` (u32) → `attnLayers = floor(L / N)`
3. si ninguna → `attnLayers = L` (comportamiento actual, modelos densos normales)

**Verificación esperada para tu caso:** 8,16 (pesos) + 2,14 (KV, bajado de 8,54) + 1,05 (overhead+mmproj) ≈ 11,35 GiB vs tus 11,99 GiB reales (~5% de error residual por estado SSM/buffers, dentro de lo razonable).

### Cambios (6 archivos)

**1. Backend — `src/optimizer.ts`**
- `GgufArch`: añadir campos `attentionLayers: number | null` y `recurrentLayers: boolean[] | null`.
- `parseGgufHeader`: leer las dos claves nuevas. Para `attention.recurrent_layers` (array GGUF) hay que **cambiar el parser** porque hoy `readGgufValue` devuelve `undefined` para arrays y solo mapea valores `typeof === 'number'`. Nuevo flujo: si la clave acaba en `.attention.recurrent_layers`, descender al array, contar `false` (= capas de atención) y guardar el array; si acaba en `.full_attention_interval`, guardar el escalar y computar `attentionLayers = floor(layers/N)`. Mantener el early-exit ampliado para incluir las nuevas claves.
- `estimateVramMiB`: en el término `kv`, usar `attnLayers = meta.attentionLayers ?? meta.layers ?? 32` en lugar de `layers`. **OJO:** `meta.layers` (total) se sigue usando para el offload fraction de pesos (`ngl/layers`) — eso no cambia (las capas SSM también tienen pesos).
- `buildEstimateResponse`: al aplicar el `offloadFraction` (línea ~371) se mantiene `meta.layers` (total). Sin cambio ahí.

**2. Backend — `src/types.ts`**
- `ModelMeta`: añadir `attentionLayers: number | null` (capas que contribuyen al KV cache; null = todas = `layers`). Documentar la distinción total vs atención.

**3. Backend — `src/router.ts`** (líneas 226-230)
- Tras `resolveModelFile`, propagar `resolved.arch.attentionLayers` → `meta.attentionLayers` si no es null.

**4. Frontend — `front/src/app/core/models/types.ts`**
- Espejo: añadir `attentionLayers: number | null` a `ModelMeta`.

**5. Frontend — `front/src/app/core/utils/vram-estimate.ts`**
- `estimateVramMiB`: usar `const attnLayers = meta.attentionLayers ?? meta.layers ?? 32` en el cálculo de `kv`. El offloadFraction sigue usando `layers` (total).
- `parseModelMeta`: como en el backend, el campo queda `null` por defecto (lo provee el GGUF vía API); no hace falta deducirlo del nombre.

**6. Frontend — `front/src/app/features/optimizer-modal/optimizer-modal.html`** (línea ~29)
- Indicador: cuando `m.attentionLayers != null && m.attentionLayers !== m.layers`, mostrar `"{{ m.attentionLayers }} / {{ m.layers }} capas de atención"` (tooltip explicando que las demás son SSM/recurrentes). Si no, mantener `"{{ m.layers }} capas"`.

### Qué NO cambia
- Modelos densos (Llama, Mistral, Gemma, Phi, DeepSeek denso…): sin `recurrent_layers` ni `full_attention_interval` → `attentionLayers = null` → `?? layers` → comportamiento idéntico al actual.
- Pesos, overhead, mmproj, `recommendParams`, `buildBreakdown` (excepto el cálculo interno de `kv` que ya baja por `estimateVramMiB`): sin cambios.
- No se toca `valueLength` ni el warning de bf16 (fuera de scope; lo mencioné aparte).

### Nota fuera de scope (mencionada, no incluida)
Tu script usa `--cache-type-k q8_0 --cache-type-v q8_0` con Qwen3.5, que según la comunidad requiere KV `bf16`. Eso es un problema potencial de tu configuración, no de la herramienta — no se toca en este fix.
